# ТЗ на переработку интерфейса калькулятора ливневого стока

## Цель

Переделать интерфейс в компактный инженерный калькулятор без длинной «простыни», с понятной последовательностью ввода, быстрым чтением результатов и сохранением расчетной логики.

## Основные требования

1. Не менять расчетное ядро без отдельной задачи.
2. Не скрывать обязательные исходные данные, но убрать визуальный шум.
3. Все данные группировать по расчетным этапам.
4. Результаты показывать в правой закрепленной колонке.
5. Второстепенные результаты и длинные проверки показывать через раскрываемые блоки.
6. Слайдеры нормативных диапазонов не держать открытыми постоянно.
7. Печатный лист оставить отдельной печатной формой.

## Новая структура страницы

### Верхняя панель

Компактная строка:

- название калькулятора;
- бейдж с редакцией СП;
- кнопка «Рассчитать»;
- кнопка «Печатный лист».

Большой декоративный hero-блок убрать.

### Левая колонка

Левая колонка содержит ввод данных и делится на пять секций:

1. Объект и место строительства.
2. Покрытия.
3. Климат и технология.
4. Дождевой расход и объем на очистку.
5. Очистные сооружения и резервуар.

### Правая колонка

Правая колонка закрепляется при прокрутке и содержит:

1. Основные результаты.
2. Проверки методики.
3. Действия.

## Секция 1. Объект и место строительства

Содержит:

- поиск населенного пункта;
- выбранное место;
- наименование объекта;
- общую площадь.

Секция должна занимать минимальную высоту.

## Секция 2. Покрытия

Оформить как плотную таблицу.

Колонки:

- покрытие;
- площадь, га;
- годовой коэффициент ψ;
- расчетный коэффициент ψ;
- признаки;
- действие удаления строки.

Признаки:

- твердое покрытие;
- мойка;
- снег;
- очистка.

Добавить кнопку «Добавить покрытие».

## Секция 3. Климат и технология

Разделить на два визуальных подблока.

### Климат

- hд теплого периода;
- hт холодного периода;
- hc талого стока за 10 часов;
- коэффициент талого стока.

### Технологические параметры

- площадь уборки снега;
- коэффициент неравномерности снеготаяния;
- площадь мойки;
- удельный расход на мойку;
- количество моек;
- коэффициент поливомоечного стока.

## Секция 4. Дождевой расход и объем на очистку

Разделить на два подблока.

### Расход в коллекторе

- площадь расчетного участка;
- q20;
- P;
- n;
- mr;
- gamma;
- Zmid;
- tcon;
- tcan;
- длина трубы;
- скорость.

### Очистка дождевого стока

- площадь на очистку;
- слой расчетного дождя ha;
- коэффициент ψ очистки;
- доля загрязненного объема.

Площадь на очистку и коэффициент ψ должны пересчитываться по покрытиям с признаком «очистка».

## Секция 5. Очистные сооружения и резервуар

Содержит:

- период переработки дождевого стока;
- период переработки талого стока;
- время отстаивания;
- технологические перерывы;
- рабочий объем резервуара;
- запас резервуара;
- режим резервуара.

## Нормативные поля

Для параметров с диапазоном показывать:

- поле числового ввода;
- единицу измерения;
- строку нормативного диапазона;
- кнопку раскрытия шкалы.

Шкала диапазона не должна быть раскрыта постоянно.

Если значение вне диапазона, поле подсвечивается и появляется требование обоснования.

Если принято крайнее значение диапазона, показывается короткое предупреждение о необходимости проверки обоснования.

## Результаты

Сразу показывать только ключевые результаты:

- общий годовой объем;
- расход дождевых вод;
- средневзвешенный ψ;
- Ky уборки снега;
- дождевой сток;
- талый сток.

Подробные результаты показывать через раскрытие:

- поливомоечные воды;
- продолжительность дождя;
- время протекания tp;
- параметр A;
- объем дождя на очистку;
- суточный объем талых вод;
- производительность очистных;
- полный объем резервуара.

## Проверки методики

В правой колонке показывать счетчики:

- ошибки;
- предупреждения;
- информационные сообщения.

Сразу показывать только три наиболее важных замечания. Полный список раскрывать по кнопке.

## Адаптивность

На широком экране — две колонки: ввод слева, результаты справа.

На планшете и телефоне — одна колонка, правый блок становится обычным блоком после ввода.

## Критерии готовности

1. Страница стала короче и плотнее.
2. Все исходные данные доступны без потери расчетной логики.
3. Покрытия оформлены таблицей.
4. Результаты и проверки находятся в правой колонке.
5. Длинные списки не раскрыты по умолчанию.
6. Печатный лист работает как раньше.
7. Тесты расчетного ядра проходят.
8. Production-сборка проходит без ошибок.
